# TabPilot 제품 개발 로드맵

> 기준: 1인 개발 또는 소규모 팀  
> 예상 기간: 약 20주  
> 제품 전략: Chrome 확장 프로그램으로 핵심 가치를 먼저 검증한 뒤, macOS 컴패니언 앱으로 시스템 수준 기능을 확장한다.

---

## 1. 제품 확장 원칙

TabPilot은 두 제품이 역할을 나누는 구조로 발전시킨다.

- **Chrome 확장 프로그램**: 탭 조회, 보호 규칙, 정리 후보 판정, `chrome.tabs.discard()` 실행
- **macOS 앱**: 시스템 메모리 압박 감지, 메뉴바 UI, 알림, 작업 모드, 로컬 통계
- **Native Messaging 연결부**: 확장 프로그램과 macOS 앱 사이에서 필요한 최소 데이터만 JSON 메시지로 교환

macOS 앱이 Chrome 탭을 외부에서 직접 제어하는 구조는 사용하지 않는다. Chrome 탭 제어는 확장 프로그램이 담당하고, macOS 앱은 시스템 상태와 사용자 인터페이스를 제공한다.

모든 단계에서 다음 원칙을 유지한다.

- 외부 서버 없이 로컬 우선
- 자동 정리는 명시적 동의 후 활성화
- 탭을 닫지 않고 네이티브 휴면 처리 우선
- 보호 여부를 점수 감점이 아닌 절대 제외 조건으로 처리
- 수집 데이터와 권한 사용 목적을 사용자에게 공개

## 3. Phase 0 — 기술 검증 및 명세 확정

**기간:** 1주

### 목표

구현 전에 Chrome API 제약과 데이터 모델을 확인하고 MVP 범위를 고정한다.

### 작업

- Manifest V3 프로젝트 및 빌드 환경 구성
- `chrome.tabs.query()`로 탭 메타데이터 조회 검증
- `Tab.lastAccessed`, `discarded`, `active`, `pinned`, `audible`, `status` 동작 검증
- `chrome.tabs.discard()` 성공·실패 조건 검증
- `chrome.system.memory` 값이 macOS에서 어떻게 표시되는지 확인
- Chrome 최소 지원 버전 결정
- 보호 도메인 서브도메인 매칭 규칙 정의
- 시크릿 창, 특수 URL, 여러 프로필 처리 방침 정의
- 설정 및 기록 저장 스키마 정의

### 완료 조건

- 실제 Chrome에서 탭 휴면 처리 PoC가 동작한다.
- 탭 정리 가능 여부를 판정하는 순수 함수와 테스트 사례가 정의된다.
- MVP 포함 기능과 제외 기능이 문서상 일치한다.

## 4. Phase 1 — Chrome 확장 프로그램 알파

**기간:** 4주

### 목표

사용자가 수동으로 안전하게 탭을 휴면 처리할 수 있는 최소 제품을 완성한다.

### 포함 기능

- 전체 탭 수, 휴면 탭 수, 정리 후보 수 표시
- 시스템 전체 메모리 참고 정보 표시
- 정리 후보 탭 목록
- 후보 사유와 제외 사유 표시
- 마지막 사용 시간 기준 설정
- 한 번에 정리할 최대 탭 수 설정
- 보호 도메인 추가·삭제
- 현재 세션에서 개별 탭 보호
- `안전 정리하기` 수동 실행
- 최근 100건 정리 기록
- 다크 모드 기반 팝업과 설정 화면

### 정리 제외 조건

- 현재 활성 탭
- 고정 탭
- 오디오 재생 탭
- 로딩 중인 탭
- 이미 휴면 상태인 탭
- 사용자가 보호한 탭 또는 도메인
- Chrome 내부 페이지와 지원하지 않는 URL

실행 직전에 탭 상태를 다시 조회하여 조건이 변경된 탭은 제외한다.

### 제외 기능

- 자동 정리
- 탭 그룹 보호
- 다운로드 상태 연계
- 중복 탭 자동 판단
- 계정 및 클라우드 동기화
- 정확한 탭별 메모리 사용량

### 품질 기준

- 정리 정책 단위 테스트
- 저장소 스키마 테스트
- 각 탭의 실패가 전체 정리를 중단하지 않는 부분 실패 처리
- ESLint, 포맷, 테스트, 프로덕션 빌드 CI
- 권한 설명 및 개인정보 처리 문서

### 완료 조건

- 보호 조건에 해당하는 탭이 테스트와 수동 검증에서 휴면 처리되지 않는다.
- 확장 프로그램 재시작 후 설정과 기록이 유지된다.
- 20~100개 탭 환경에서 팝업과 정리 실행이 안정적으로 동작한다.

## 5. Phase 2 — Chrome 확장 프로그램 베타

**기간:** 3주

### 목표

실사용 피드백을 받아 안전성과 사용성을 검증한다.

### 작업

- 10~30명 규모 비공개 테스트
- 최초 실행 안내와 보호 프리셋 선택 화면
- 정리 전 미리보기 및 실행 결과 화면
- 전체 탭 목록을 별도 페이지로 분리
- 도메인별 보호 규칙 편집 개선
- 정리 취소 대신 휴면 탭 재활성화 진입점 제공
- 오류 기록과 진단 정보 내보내기
- 키보드 접근성과 화면 읽기 기본 대응
- 성능 및 서비스 워커 수명 주기 점검

### 검증 지표

- 첫 정리 완료율
- 정리 실행당 성공 및 실패 탭 수
- 정리 후 5분 이내 재활성화 비율
- 사용자가 직접 추가한 보호 도메인 수
- 잘못 정리되었다고 판단한 사례
- 팝업 초기 표시 시간

분석 서버는 사용하지 않는다. 베타 피드백과 사용자가 명시적으로 내보낸 진단 정보로 검증한다.

### 완료 조건

- 데이터 손실 또는 중요 탭 오정리와 관련된 치명적 문제가 없다.
- 주요 오류 상황에 사용자에게 이해 가능한 결과가 표시된다.
- 공개 배포에 필요한 문서와 스토어 자료가 준비된다.

## 6. Phase 3 — Chrome Web Store 공개

**기간:** 2주

### 목표

Chrome 확장 프로그램을 독립 제품으로 공개하고 핵심 수요를 검증한다.

### 작업

- Chrome Web Store 등록 자료 제작
- 권한 최소화 최종 점검
- 개인정보 처리방침 공개
- 스크린샷, 아이콘, 설명문 작성
- GitHub 저장소와 이슈 템플릿 공개
- 버전 및 변경 기록 정책 수립
- 베타 채널과 안정 채널 운영 방식 결정

### 공개 후 관찰 기간

최소 4주 동안 다음 항목을 관찰한다.

- 설치 후 실제 정리 기능을 사용하는지
- 보호 규칙에 대한 요구가 많은지
- 시스템 메모리 기반 기능을 원하는 사용자가 충분한지
- macOS 메뉴바 앱을 별도로 설치할 의향이 있는지

### macOS 앱 진행 기준

아래 조건 중 다수가 확인되면 macOS 앱 개발을 진행한다.

- Chrome 단독 제품의 반복 사용자가 존재한다.
- 단순 시간 기준보다 메모리 압박 기반 정리 요구가 반복된다.
- 메뉴바 상태 확인과 자동화 요구가 반복된다.
- 네이티브 앱 설치 부담보다 통합 기능의 가치가 크다는 피드백이 확인된다.

## 7. Phase 4 — macOS 연동 기술 검증

**기간:** 2주

### 목표

macOS 앱 개발 전에 설치, 통신, 샌드박스, 배포 리스크를 제거한다.

### 권장 기술

- Swift
- SwiftUI
- `MenuBarExtra` 기반 메뉴바 UI
- macOS 시스템 메모리 압박 이벤트와 시스템 상태 API
- Chrome Native Messaging
- 로컬 저장소

### PoC 범위

- 메뉴바에서 메모리 상태 표시
- macOS 앱과 Chrome 확장 프로그램 연결 상태 확인
- macOS 앱에서 정리 요청 전송
- 확장 프로그램에서 후보 수와 정리 결과 반환
- 앱이 종료되거나 연결되지 않아도 확장 프로그램이 독립적으로 작동
- Native Messaging host 등록 및 제거 절차 검증
- 앱 서명, 공증, 업데이트 및 제거 절차 검토

### 주요 리스크

- Native Messaging 사용을 위해 확장 프로그램에 `nativeMessaging` 권한이 추가된다.
- macOS에서 host manifest와 실행 파일의 설치 위치를 관리해야 한다.
- Chrome Web Store 확장 ID가 Native Messaging 허용 목록과 정확히 일치해야 한다.
- Mac App Store 샌드박스와 설치 방식은 별도 검증이 필요하다.

초기 macOS 배포는 개발자 서명과 공증을 거친 직접 배포를 우선 검토하고, Mac App Store 배포는 기술 검증 후 결정한다.

### 완료 조건

- 서명된 macOS 앱과 스토어용 확장 프로그램이 양방향 통신한다.
- 설치 및 제거 후 Native Messaging 설정이 정상적으로 정리된다.
- 연결이 끊겨도 Chrome 확장 프로그램 핵심 기능에 영향이 없다.

## 8. Phase 5 — macOS 메뉴바 앱 베타

**기간:** 5주

### 목표

Chrome 확장 프로그램을 보조하는 시스템 수준 작업 환경 관리 기능을 제공한다.

### 포함 기능

- 메뉴바 메모리 상태 표시
- 정상, 주의, 압박 상태 표시
- Chrome 탭 수, 휴면 탭 수, 정리 후보 수 표시
- 메뉴바에서 안전 정리 요청
- 저메모리 상태 알림
- 사용자 승인 후 긴급 정리
- 집중 작업용 보호 모드
- 정리 기록과 메모리 상태의 로컬 타임라인
- Chrome 확장 프로그램 설치 및 연결 안내

### 자동화 원칙

- 초기에는 알림과 추천만 제공
- 사용자가 별도로 활성화한 경우에만 자동 정리
- 자동 실행 직전에도 확장 프로그램이 보호 조건을 재검증
- 네이티브 앱은 탭 URL이나 제목 전체를 요구하지 않고 집계와 명령 위주로 통신
- 민감한 탭 메타데이터를 macOS 앱에 보낼 때는 목적과 보관 기간을 명시

### 완료 조건

- 메모리 압박 이벤트에서 중복 알림이나 반복 정리가 발생하지 않는다.
- 앱과 확장 프로그램 버전이 달라도 호환성 오류가 안전하게 처리된다.
- 자동 정리는 명시적 동의 없이 활성화되지 않는다.
- 앱 자체의 CPU 및 메모리 사용량이 상시 실행에 적합하다.

---

## 9. Phase 6 — 통합 제품 안정화 및 배포

**기간:** 3주

### 목표

Chrome 단독 사용자와 macOS 통합 사용자 모두에게 안정적인 제품을 제공한다.

### 작업

- 확장 프로그램과 앱의 메시지 프로토콜 버전 관리
- 연결 상태 및 호환 버전 안내
- 설치, 업데이트, 제거 플로우 자동화
- 장애 진단 번들 내보내기
- 코드 서명과 공증 자동화
- 업데이트 채널 운영
- 온보딩 및 지원 문서 작성
- 보안 검토와 위협 모델 작성

### 배포 형태

1. **Chrome 단독 버전**
   - Chrome Web Store에서 설치
   - 수동 안전 정리와 기본 보호 기능 제공

2. **TabPilot for Mac**
   - macOS 메뉴바 앱 설치
   - 기존 Chrome 확장 프로그램과 연결
   - 시스템 메모리 압박, 알림, 작업 모드 제공

macOS 앱은 확장 프로그램의 필수 구성 요소가 아니라 선택형 고급 기능으로 유지한다.

## 10. 이후 확장 후보

통합 제품의 안정성이 확인된 뒤 아래 기능을 검토한다.

- 프로젝트별 탭 워크스페이스
- 회의, 코딩, 리서치 작업 모드
- 업무 종료 전 탭 세션 아카이브
- Safari 확장 프로그램
- Chromium 기반 Edge 지원
- 여러 브라우저의 메모리 상태 통합 표시
- 사용자 선택형 iCloud 설정 동기화

계정 시스템, 외부 분석 서버, 클라우드 동기화는 명확한 사용자 가치가 검증되기 전에는 도입하지 않는다.

## 11. 권장 첫 출시 범위

첫 공개 버전은 다음 한 문장으로 설명할 수 있어야 한다.

> 보호된 작업 탭은 건드리지 않고, 오래 사용하지 않은 Chrome 탭을 사용자가 확인한 뒤 안전하게 휴면 처리한다.

메모리 압박 기반 자동화와 macOS 메뉴바 기능은 이 핵심 경험이 검증된 이후에 추가한다.
